# AgentTeams 客户端化改造技术方案（保持 B/S 架构不变）

> 目标：面向个人提效场景，在不破坏现有 Web B/S 主架构的前提下，新增本地文件处理与本地知识库能力。

---

## 1. 背景与目标

当前 `AgentTeams` 已具备完整的 B/S 形态（`frontend` + `FastAPI` + `WebSocket`）。  
你的诉求不是替换现有架构，而是引入“客户端能力”以支持：

- 本地文件深度处理（目录扫描、增量处理、批量解析）
- 本地知识库（索引、检索、引用回溯）
- 个人设备能力（后续可扩展系统通知、剪贴板、定时任务等）

### 1.1 改造原则

- **B/S 主链路不改**：保留现有 `POST /api/tasks` + `WS /ws` 交互模式
- **客户端能力可插拔**：未安装客户端时系统照常可用
- **本地优先**：本地文件与索引默认不上传
- **最小侵入后端**：通过独立适配层接入，不改动 `team-lead` 主编排逻辑

---

## 2. 总体方案（推荐）

采用 **“双通道架构”**：

1. **原有云端/本地后端通道（保留）**
   - 继续承载任务编排、多专家协作、流式输出
2. **新增本地客户端能力通道（新增）**
   - 在用户设备上运行一个轻量客户端服务（AgentTeams Client Runtime）
   - 专门负责本地文件与知识库工具调用

### 2.1 逻辑架构

```text
┌────────────────────────────────────────────────────────┐
│                    现有 Web 前端 (Vue)                 │
│      任务输入 / 会话列表 / 流式输出 / 场景表单          │
└───────────────────────┬────────────────────────────────┘
                        │ HTTP/WS（保持不变）
┌───────────────────────▼────────────────────────────────┐
│                 现有后端 (FastAPI + team-lead)         │
│  Engine / MessageBus / SkillRegistry / Memory / DB     │
│  + 新增 Local Capability Adapter（薄层）               │
└───────────────┬────────────────────────────────────────┘
                │ localhost RPC（仅在启用客户端时）
┌───────────────▼────────────────────────────────────────┐
│         AgentTeams Client Runtime（新增，本机）         │
│  File Service | Indexer | Retriever | Local Store      │
│  (Tauri/Electron UI 可选 + 常驻后台服务)               │
└────────────────────────────────────────────────────────┘
```

---

## 3. 关键设计决策

### 3.1 为什么不把“客户端能力”直接塞进后端

- 后端可部署在远端服务器时，无法直接访问用户本地文件
- 本地索引过程（尤其大文件、批量目录）更适合在终端执行
- 权限边界更清晰：本地数据只由本机 Runtime 管理

### 3.2 为什么不是“重写成纯桌面应用”

- 你已有成熟 B/S 流程（前后端、WebSocket、任务编排）  
- 纯重写成本高且会破坏既有能力与使用习惯
- 双通道方案可渐进落地，风险更低

### 3.3 客户端技术选型建议

优先级建议：

- **首选：Tauri + Python Sidecar**
  - 客户端壳体轻量（资源占用小）
  - 可复用你现有 Python 能力（文档解析、向量化、索引）
- **备选：Electron**
  - 生态成熟，开发门槛低
  - 但体积与内存占用偏大

结论：若目标是“个人长期常驻提效”，更建议 `Tauri` 方向。

---

## 4. 模块拆分设计

新增以下模块（尽量独立于现有主链路）：

### 4.1 后端新增：`Local Capability Adapter`

建议目录：

- `src/tools/local_client_adapter.py`
- `src/tools/local_kb_tool.py`
- `src/tools/local_file_tool.py`

职责：

- 探测本机 Runtime 是否可用（健康检查）
- 封装本地能力调用（检索、索引、文件读取）
- 统一降级策略（Runtime 不可用时返回可解释提示）

### 4.2 客户端新增：`AgentTeams Client Runtime`

建议职责边界：

- `file_service`：目录扫描、文件读取、格式转换（txt/md/pdf/docx）
- `index_service`：分块、向量化、增量索引、重建策略
- `retrieval_service`：向量检索 + 关键词检索 + 混合召回
- `permission_service`：目录授权白名单
- `local_store`：SQLite + 向量库（如 Chroma/FAISS）

---

## 5. 接口契约（后端 ↔ 客户端）

通信建议：`localhost` HTTP（短期）-> gRPC（中期可选）

### 5.1 健康检查

- `GET /health`
- 返回：`{ "status": "ok", "version": "x.y.z" }`

### 5.2 索引任务

- `POST /kb/index`
- 入参：`paths[]`, `collection`, `reindex`, `file_types[]`
- 出参：`task_id`, `accepted`

### 5.3 检索

- `POST /kb/search`
- 入参：`query`, `collection`, `top_k`, `filters`
- 出参：`hits[]`（含片段、来源路径、score）

### 5.4 文件读取

- `POST /file/read`
- 入参：`path`, `start`, `end`, `mode`
- 出参：`content`, `mime`, `meta`

### 5.5 权限管理

- `POST /permissions/grant`
- `GET /permissions/list`

说明：仅允许访问显式授权目录，避免全盘读写风险。

---

## 6. 对现有 B/S 架构的影响评估

### 6.1 不变部分

- 前端主交互：会话、消息流、模型切换、任务提交
- 后端主流程：`Engine -> team-lead -> MessageBus -> WS`
- 数据主库：`data/agent_teams.db`（会话消息）

### 6.2 增量变更部分

- 新增本地能力工具（仅在任务命中相关场景时调用）
- 新增“客户端在线状态”提示（前端可选显示）
- 场景模板可新增“本地知识库问答”类型（YAML 配置）

### 6.3 降级策略

- 未安装客户端：自动回落到现有 B/S 行为
- 客户端离线：提示“本地能力不可用”，任务仍可继续
- 索引失败：保留错误原因与重试指引，不阻断主任务

---

## 7. 安全与隐私设计

- Runtime 仅监听 `127.0.0.1`
- 每次连接带短期 token（后端启动后动态协商）
- 目录白名单授权，默认拒绝
- 关键操作审计日志（索引、读取、删除索引）
- 索引库默认本地存储，不主动上传

---

## 8. 分阶段实施计划

### Phase 1（1 周）：能力打通（最小可用）

- 建立 Runtime 最小服务：`/health`、`/file/read`、`/kb/search`（可先假数据）
- 后端完成 `local_client_adapter` 与调用兜底
- 前端增加“客户端状态”提示（可选）

验收标准：

- 不安装客户端时，系统行为与现在一致
- 安装客户端后，可完成一次本地文件检索问答

### Phase 2（1-2 周）：本地知识库可用

- 接入索引链路：`/kb/index`（目录 -> 分块 -> 向量）
- 增量更新与重建策略
- 场景模板新增“本地知识问答/文档总结”

验收标准：

- 对指定目录建立索引后，检索结果可稳定返回来源路径与片段

### Phase 3（1 周）：产品化与稳定性

- 权限白名单 UI
- 索引任务进度与失败重试
- 性能优化（缓存、并发、批处理）

验收标准：

- 10k+ 文档片段规模下，检索响应和体验稳定

---

## 9. 风险与应对

- **风险 1：本地解析格式复杂（PDF/扫描件）**
  - 应对：先支持 txt/md/docx/pdf（文本型），OCR 放后续
- **风险 2：索引耗时长**
  - 应对：异步任务 + 断点续跑 + 增量索引
- **风险 3：客户端生命周期管理**
  - 应对：提供托盘常驻 + 自启动开关 + 健康探针重连
- **风险 4：后端工具调用阻塞**
  - 应对：工具调用设超时与熔断，失败自动降级

---

## 10. 与当前代码的映射建议（便于落地）

- 后端入口与编排：继续使用 `src/main.py`、`src/agents/team_lead.py`
- 工具扩展点：在 `src/tools/` 下新增本地能力适配器
- 技能触发：在 `data/skills/` 增加 `local_kb`、`local_file` 技能定义
- 场景扩展：在 `data/scenes/` 增加本地知识库相关场景模板

建议优先改造为“可选能力”，避免一次性改动过大。

---

## 11. 结论

这套方案的核心是：**保留 B/S 主架构，新增客户端能力层**。  
它同时满足你的三个目标：

- 个人提效（本地文件与知识库）
- 架构连续性（不推倒重来）
- 可持续演进（后续可扩展更多本地能力）

从工程风险和投入产出比看，这是当前最稳妥且可快速见效的路线。
