# AgentTeams 功能演进规划 — 面向个人提效

> 本文档基于当前系统能力，围绕"让普通用户（非技术背景）真正用起来"这个目标，进行三轮递进式思考。

---

## 第一轮思考：从用户视角出发 — 什么值得做？

### 1.1 核心问题

当前系统有一个根本矛盾：**能力很强，但用户不知道怎么用。**

一个小白用户打开系统，看到一个输入框，他的第一反应是"我该输入什么？"。这不是技术问题，是产品问题。所以在想"加什么功能"之前，先想清楚：**用户在什么场景下会打开这个系统？**

### 1.2 真实使用场景分析

按频率和价值两个维度画一个矩阵：

```
价值高 ↑
        │  ③方案决策     ①公文写作
        │  ④竞品调研     ②汇报总结
        │
        │  ⑥节日问候     ⑤信息搜索整理
        │  ⑦朋友圈文案   ⑧知识学习
        └──────────────────────→ 频率高
```

**第一象限（高频+高价值）**— 最该做的：
- **公文/通知/方案写作**：几乎每天都在写，痛点极大（格式要求严、要素多、改来改去）
- **汇报材料整理**：周报月报述职，重复性高但每次都花时间

**第二象限（低频+高价值）**— 做了有亮点：
- **方案评估/决策辅助**：不常用，但每次用都能省大量时间
- **竞品/行业调研**：需要的时候很急，手动搜+整理至少半天

**第三象限（高频+低价值）**— 锦上添花：
- **日常信息查询**：快速问答，当前已支持
- **知识学习辅助**：有用但不紧急

**第四象限（低频+低价值）**— 暂时不做：
- **朋友圈文案**、**节日群发**：用户自己就能做

### 1.3 第一轮结论

> **优先做"写作全流程"和"研究全流程"，因为这两个场景的用户痛点最大、使用频率最高，且当前系统架构天然适配。**

---

## 第二轮思考：怎么做？— 从"能用"到"好用"

### 2.1 当前系统的用户体验瓶颈

站在小白用户的角度走一遍流程，会发现几个断点：

| 断点 | 表现 | 原因 |
|---|---|---|
| **不知道能做什么** | 用户面对空白输入框发呆 | 没有引导和示例 |
| **不知道怎么描述** | 用户说"帮我写个东西"，太模糊 | 缺少结构化引导 |
| **等待时间焦虑** | 专家协作需要 30-60 秒 | 过程不透明，用户不知道在干嘛 |
| **结果不满意没法改** | 只能重新提问 | 缺少迭代修改机制 |
| **好的结果没法用** | 复制粘贴到 Word/邮件 | 缺少导出能力 |

### 2.2 针对每个断点的解决思路

#### 断点 1：不知道能做什么 → **场景卡片 + 快捷入口**

用户进入系统后，不是看到空白，而是看到一组**场景卡片**：

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  📝 公文写作  │  │  📊 调研分析  │  │  🤔 决策辅助  │
│  通知/方案/总结 │  │  竞品/行业/政策 │  │  利弊/选型/评估 │
└─────────────┘  └─────────────┘  └─────────────┘
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  📖 学习助手  │  │  ✈️ 生活规划  │  │  💬 沟通协助  │
│  概念/面试/读书 │  │  旅行/购物/健康 │  │  邮件/消息/催办 │
└─────────────┘  └─────────────┘  └─────────────┘
```

**关键设计决策：** 卡片不是装饰，而是**快捷入口**。点击后直接弹出该场景专属的信息收集表单（复用现有的 info_request 卡片机制），而不是让用户自己组织语言。

例如点击"公文写作"后展开子类型选择：

```
请选择文体类型：
[ 会议通知 ] [ 工作方案 ] [ 请示报告 ] [ 述职总结 ] [ 通报 ] [ 其他... ]
```

选择"会议通知"后，自动弹出信息卡片收集时间、地点、参会人等。

**这里的核心洞察：** 不是让 AI 猜用户要什么，而是用结构化交互把需求确定下来，再交给 AI 执行。**把"创造性理解"变成"结构化填表"，小白用户才能用得顺。**

#### 断点 2：不知道怎么描述 → **模板 + 引导式提问**

对于每个场景，预设一组**提问模板**：

- 写作类：「帮我写一篇【文体】，主题是【主题】，用于【场景】，面向【受众】」
- 调研类：「帮我调研【对象】的【方面】，重点关注【关注点】」
- 决策类：「我在纠结【选项A】还是【选项B】，我的情况是【背景】」

用户可以直接点击模板，系统把【占位符】替换成输入框，用户只需填空。

**实现方式思考：** 这不需要改后端逻辑。模板本质上是前端的输入辅助，最终还是拼成一段自然语言发给后端。可以用 JSON 配置文件定义模板，前端读取渲染。

#### 断点 3：等待时间焦虑 → **过程可视化增强**

当前已有"正在分析任务"的状态提示，但不够。用户需要看到：

```
✅ 任务理解完成（2s）
✅ 已分配 2 位专家：行业分析师、风险评估师（3s）
🔄 行业分析师 正在联网搜索 "2026年新能源汽车市场"...
⏳ 风险评估师 等待协作...
```

这个进度信息当前后端已经在发（status_update），问题在于前端展示不够直观。可以考虑：
- 用一个**时间线/步骤条**替代当前的纯文本状态
- 每个专家有独立的进度指示器
- 搜索结果有简要预览（"找到 5 条相关结果"）

#### 断点 4：结果不满意没法改 → **迭代对话 + 局部修改**

这是当前系统最大的体验缺口。用户看到结果后的常见反应：

- "整体不错，但第二段太啰嗦了" → 需要**局部修改**
- "换个更正式的语气" → 需要**风格调整**
- "再加一段关于预算的内容" → 需要**追加内容**

**实现思路：**

方案 A：在最终回答下方增加一个"继续对话"入口，用户的后续输入会携带之前的回答作为上下文，发起新的任务。本质是一个**带上下文的追问**。

方案 B：更精细的做法 — 允许用户选中回答中的某一段文字，弹出"修改此段"的浮动按钮，点击后只对选中段重新生成。

方案 A 实现简单，方案 B 体验更好。建议先做 A，验证需求后再升级到 B。

#### 断点 5：好的结果没法用 → **一键导出**

用户写完一篇通知，需要：
- 复制纯文本（粘贴到 Word/WPS）
- 复制 Markdown（粘贴到飞书/语雀）
- 下载为 .docx 文件（直接打印/分发）

在最终回答的右上角增加导出按钮组：`[ 📋 复制 ] [ 📄 Word ] [ 📝 Markdown ]`

### 2.3 第二轮结论

> **不急着加新的 AI 能力，先解决"用户不会用"和"用完不方便"的问题。场景卡片（入口）+ 信息表单（收集）+ 过程可视化（等待）+ 迭代修改（打磨）+ 一键导出（交付）—— 这五步形成完整的用户体验闭环。**

## 第三轮思考：落地路径 — 分阶段、可验证

### 3.1 阶段划分

#### 阶段一：让写作场景跑通（1-2 周）

**目标：** 一个完全不懂 AI 的办公室文员，能用这个系统写出一篇合格的会议通知。

需要做的事：

1. **场景模板系统**
   - 定义一套 JSON/YAML 格式的场景模板配置
   - 每个模板包含：场景名、图标、子类型列表、每个子类型对应的信息收集字段
   - 前端渲染为卡片入口
   - 点击后自动触发 info_request 流程（复用现有卡片机制）
   - 用户填完后拼成结构化的任务描述发给后端

2. **写作类 prompt 专项优化**
   - 当前的通用 prompt 不够针对写作场景
   - 需要为"公文写作"这个大类设计专门的专家组合策略：
     - 文体格式专家（确保格式合规）
     - 内容撰写专家（确保内容到位）
   - 在技能系统中新增"公文写作"技能，包含常见公文格式要求

3. **复制功能**
   - 最终回答右上角加"复制"按钮（纯文本 + Markdown 两种）
   - 技术上很简单但用户体验提升很大

**验证标准：** 找 3 个非技术人员，让他们独立完成"写一篇会议通知"的任务，全程不需要指导。

#### 阶段二：让研究场景可用（1-2 周）

**目标：** 用户给一个调研主题，系统能产出一份结构化的调研报告。

需要做的事：

1. **调研场景模板**
   - 竞品分析模板：收集产品名、对比维度、关注点
   - 行业研究模板：收集行业名、时间范围、分析角度
   - 政策解读模板：收集政策链接或名称、关注影响范围

2. **搜索能力增强**
   - 当前每个专家只能搜索一次，复杂调研可能需要多轮搜索
   - 考虑允许专家在第二轮协作时补充搜索（当前只在第一轮搜索）
   - 搜索结果的引用标注（来源链接），提升报告可信度

3. **报告结构化输出**
   - 调研报告需要有固定结构：摘要 → 背景 → 核心发现 → 分析 → 建议 → 参考来源
   - 在 prompt 中为调研类任务增加输出格式要求
   - 前端对报告类内容增加目录导航

**验证标准：** 用"分析 2026 年 AI Agent 赛道的主要玩家和趋势"做测试，产出的报告能作为内部分享材料直接使用。

#### 阶段三：交互体验打磨（1-2 周）

**目标：** 系统从"能用"升级到"好用"。

需要做的事：

1. **迭代修改能力**
   - 最终回答下方增加"继续优化"输入框
   - 后续输入自动携带之前的完整回答作为上下文
   - prompt 中增加"用户要求修改，请基于原文调整"的指令

2. **过程可视化升级**
   - 将 status_update 消息渲染为步骤条/时间线
   - 专家头像 + 实时状态指示（搜索中/思考中/协作中）
   - 整体耗时预估（"预计还需 20 秒"）

3. **历史记录增强**
   - 按场景分类查看历史（"我之前写过的通知""我做过的调研"）
   - 支持收藏/置顶常用结果
   - 支持从历史记录中"再来一次"（复用之前的参数重新生成）

4. **导出能力**
   - Word 导出（后端用 python-docx 生成）
   - PDF 导出（后端用 weasyprint 或前端用 html2pdf）
   - 纯文本/Markdown 复制（前端实现）

**验证标准：** 用户满意度调查，目标是 80% 的用户认为"比直接用 ChatGPT 方便"。

#### 阶段四：能力扩展（持续迭代）

在前三个阶段跑通后，再考虑加新能力：

| 能力 | 实现方式 | 前置条件 |
|---|---|---|
| 日历集成 | 新增技能，调用系统日历 API | macOS 权限授权 |
| 本地文档批量读取 | 扩展 file_reading 技能，支持目录扫描 | 文件权限 |
| Excel 分析 | 新增技能，用 pandas 处理 | 后端增加数据处理模块 |
| 图片生成 | 新增技能，调用 DALL-E / SD API | 配置图片生成 API |
| 邮件发送 | 新增技能，类似 iMessage | 配置 SMTP |
| 语音输入 | 前端增加语音识别 | 浏览器 Web Speech API |

### 3.2 技术架构层面的思考

当前架构对上述规划的支撑情况：

**已经具备的基础（可直接复用）：**
- ✅ 技能注册表（YAML 定义）→ 新技能只需加配置文件
- ✅ 信息卡片机制 → 场景模板的交互层可直接复用
- ✅ 专家动态创建 → 不同场景的专家组合只需调整 prompt
- ✅ 流式输出 → 用户体验已有基础
- ✅ WebSocket 实时通信 → 过程可视化有通道

**需要新增的基础设施：**
- ❌ 场景模板配置系统（前端读取 + 渲染）
- ❌ 上下文追问机制（当前每个任务是独立的）
- ❌ 文件导出服务（后端生成 Word/PDF）
- ❌ 前端组件：场景卡片、步骤条、导出按钮、修改输入框

**需要重点优化的：**
- ⚠️ prompt 工程：通用 prompt → 场景化 prompt（写作、调研、决策各一套）
- ⚠️ 专家分配策略：目前完全由 LLM 决定，可能需要场景化的默认专家配置
- ⚠️ 上下文管理：迭代修改需要维护对话上下文，当前任务之间是隔离的

### 3.3 一个关键的产品判断

三轮思考后，我认为最重要的一个认知是：

> **这个系统的核心竞争力不是"AI 更聪明"，而是"流程更顺畅"。**
>
> 用户用 ChatGPT 也能写通知，但他需要自己想怎么提问、自己判断结果好不好、自己复制到 Word 里。而 AgentTeams 的价值在于：
>
> 1. **不用想怎么提问** — 场景卡片 + 信息表单帮你结构化
> 2. **不用担心质量** — 多专家协作 + 领域核查双重保障
> 3. **不用手动整理** — 一键导出到你要的格式
>
> 所以优先级永远是：**流程顺畅 > 功能数量 > AI 能力。**

### 3.4 第三轮结论

> **四个阶段，每个阶段 1-2 周，3 个月内完成从"技术原型"到"个人生产力工具"的转变。核心策略：先做深一个场景（写作），验证用户体验闭环，再横向复制到其他场景。不要贪多，不要追求技术炫酷，始终以"小白用户能独立完成任务"为验证标准。**
